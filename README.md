## ⚡ 핵심 기술 선택 근거

### 1. Kafka Choreography Saga (분산 트랜잭션 대체)

**문제:** 서비스 간 데이터 정합성을 2PC(Two-Phase Commit) 없이 보장해야 함  
**선택:** Choreography Saga + Transactional Outbox Pattern  
**결과:** 각 서비스는 로컬 트랜잭션만 실행, 실패 시 보상 이벤트로 롤백

```
접수 등록 (Admin)
  → VISIT_REGISTERED 이벤트 발행 (Outbox)
  → Clinical: visit_clinical_status INSERT (WAITING)

검사오더 + SOAP 완성 (Clinical)
  → DIAGNOSTIC_ORDER_SUBMITTED 발행
  → Support: 워크리스트 생성

검사 완료 (Support)
  → DIAGNOSTIC_ORDER_COMPLETED 발행
  → Clinical: 최종오더 가능 상태로 전환

최종오더 확정 (Clinical)
  → BILLING_REQUESTED 발행
  → Admin: 청구서 자동 생성
  → BILLING_COMPLETED / BILLING_FAILED 발행
```

### 2. Oracle 다중 스키마 + Flyway

**문제:** 서비스별 DB 독립성 유지 (MSA 원칙 — 서비스 간 JOIN/FK 금지)  
**선택:** Oracle XE 21c 단일 인스턴스 + 4개 독립 스키마  
**이유:** 운영 비용을 최소화하면서 스키마 레벨 격리로 MSA 경계 구현  
**Flyway:** `@GeneratedValue` + Oracle IDENTITY 컬럼, `GENERATED BY DEFAULT ON NULL AS IDENTITY` 패턴 적용

### 3. React Query (TanStack Query v5)

**문제:** Context API 기반 클라이언트 상태 관리의 서버 데이터 캐시 미지원  
**선택:** React Query — 서버 상태와 클라이언트 상태 분리  
**결과:**
- `stale-while-revalidate` 전략으로 불필요한 API 호출 최소화
- `invalidateQueries` 기반 낙관적 업데이트
- 서버 모드(IAM 로그인) / 로컬 모드(데모) 이중 소스 처리

### 4. JWT + RBAC (HttpOnly Cookie)

**문제:** localStorage 저장 시 XSS 취약점  
**선택:** AccessToken 메모리 보관 + RefreshToken HttpOnly Cookie  
**역할 체계:** SYS > ADMIN > DOC > NUR (Spring PreAuthorize 기반)

```
예약 등록 (RESERVED)
    │
    ▼

    ▼

    │  SOAP 작성 + 검사 오더 요청
    │  DIAGNOSTIC_ORDER_SUBMITTED →   Support: 워크리스트 생성
    ▼
검사 완료 (EXAM_DONE)                 ← Support Service
    │  DIAGNOSTIC_ORDER_COMPLETED →   Clinical: 최종오더 가능 상태
    ▼
최종 오더 확정                         ← Clinical Service
    │  BILLING_REQUESTED →            Admin: 청구서 자동 생성
    ▼
청구 완료 (COMPLETED)                 ← Admin Master Service
    │
    ▼
원무직원 접수목록에서 삭제 가능
```

## 🔑 주요 설계 결정 (ADR 요약)

### ADR-001: 서비스 간 통신 전략
- **결정:** 동기(REST) vs 비동기(Kafka) 이분 원칙
- **동기:** IAM 인증 확인, Clinical→Support 검사결과 조회
- **비동기:** 도메인 이벤트 전파 (접수→진료→검사→청구 전 체인)
- **근거:** 장애 격리 — Support 다운 시 Clinical 영향 없음

### ADR-002: Outbox Pattern 도입
- **결정:** 각 서비스 DB에 `outbox_event` 테이블 별도 운영
- **근거:** Kafka 발행과 DB 저장의 원자성 보장 (2PC 없이)
- **구현:** `@Scheduled(fixedDelay=2000ms)` Publisher가 NEW → PUBLISHED 전환

### ADR-003: Oracle 다중 스키마 vs 다중 DB
- **결정:** Oracle XE 단일 인스턴스 + 4개 스키마
- **근거:** 포트폴리오 환경에서 운영비 최소화, 스키마 격리로 MSA 경계 유지
- **제약:** 서비스 간 스키마 직접 JOIN 금지 (코드 레벨 강제)
